## 0. 최우선 결론

현재 수행평가 탐구엔진은 보고서 생성 틀과 UI 구조는 이미 잡혀 있다.

지금 해야 할 일은 다음이다.

```text
1. UI 구조 변경 금지
2. 보고서 생성 틀 변경 금지
3. 1~8번 흐름 변경 금지
4. major_search_fast_guard.js 재추가 금지
5. 학과 검색 UI를 후보 카드 방식으로 바꾸기 금지
6. 현재는 학과 데이터부터 잠그는 단계
7. 3번 교과 개념 / 4번 후속 연계축은 학과 데이터 안정화 후 진행
8. 패치는 필요한 파일만 최소 단위로 제공
9. 사용자가 직접 코드 수정하지 않도록 zip 또는 교체용 파일로 제공
10. 파일명은 짧은 영문명으로 제공
````

가장 중요한 문장:

```text
지금은 교과 연결성 검수가 아니라 학과 데이터 잠금 단계다.
학과 검색 → 대표 학과 → 관련 학과 묶음 → 빠른 비교 학과가 먼저 안정화되어야 한다.
```

## 1. 새 채팅에 업로드할 파일

새 채팅에는 아래 파일을 업로드한다.

```text
1. 최신 keyword-engine.zip
```

현재 이 채팅에서는 ChatGPT 파일 다운로드/업로드 상태 동기화 오류가 발생했다.

오류 메시지:

```text

/mnt/data/v84fix.zip의 업로드 상태를 가져오지 못했습니다
/mnt/data/v84.zip의 업로드 상태를 가져오지 못했습니다
```

따라서 v84 파일은 생성 시도는 했지만, 사용자가 정상 다운로드하지 못했다.
즉, 사용자의 실제 로컬 keyword-engine 상태는 v84가 아니라 v83 적용 상태일 가능성이 높다.

새 채팅에서는 반드시 업로드된 최신 keyword-engine.zip 내부를 먼저 확인해야 한다.

## 3. fast guard 상태

사용자는 major_search_fast_guard.js를 삭제한 상태다.

절대 다시 추가하면 안 된다.

확인 기준:

```text
index.html에서 major_search_fast_guard.js 로드가 없어야 함
window.__MAJOR_SEARCH_FAST_GUARD_VERSION 값이 없어야 함
검색 UI가 후보 카드 방식으로 바뀌면 안 됨
```

금지 파일:

```text
keyword-engine/assets/major_search_fast_guard.js
```

이 파일은 과거에 검색 속도 개선 목적으로 추가했지만, 아래 문제를 만들었기 때문에 폐기했다.

```text
- 별도 후보 카드 UI 생성
- 학과를 한 번 더 클릭해야 하는 구조 발생
- 4번 이후 다시 학과 검색 시 멈춤
- 3번/4번 상태 초기화와 충돌
- 기존 native major_engine_helper 흐름과 충돌
```
