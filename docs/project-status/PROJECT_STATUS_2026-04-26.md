## 2. 수행평가 탐구엔진의 핵심 철학

이 프로젝트는 단순 키워드 추천기가 아니다.

엔진의 핵심은 아래 흐름이다.

```text
교과 개념
→ 추천 키워드
→ 후속 연계축
→ 도서
→ 보고서 전개 방식
→ 보고서 관점
→ 보고서 라인
→ MINI/Worker 생성 결과
```

학생이 학과를 먼저 입력하더라도, 보고서는 학과 홍보문이 아니라 **교과 개념에서 출발한 수행평가 탐구 보고서**가 되어야 한다.

따라서 학과는 다음 역할을 한다.

```text
1. 교과 개념의 우선순위를 조정한다.
2. 후속 연계축의 우선순위를 조정한다.
3. 도서 추천의 직접 일치/확장 참고 여부를 조정한다.
4. 최종 보고서에서 전공 관점을 덧입힌다.
```

학과가 교과 개념을 밀어내면 안 된다.
하지만 학과가 바뀌었는데 결과가 동일해도 안 된다.

## 3. 횡단 평가와 종단 연결 기준

다음 작업자가 반드시 알아야 하는 개념이다.

### 3-1. 횡단 평가

횡단 평가는 같은 학년 또는 같은 시기 안에서 교과 간 연결성이 보이는 구조다.

예시:

```text
통합과학에서 폭염주의보를 다룸
→ 통합사회에서 취약 계층·생활환경 문제와 연결
→ 수학에서 자료 비교·그래프 해석과 연결
→ 정보에서 입력값·조건문·판단 기준과 연결
```

횡단 평가는 학생이 한 개념을 여러 과목에서 어떻게 다르게 바라보는지를 보여준다.

보고서에서는 다음처럼 반영된다.

```text
자료 해석
사회적 의미
수치 비교
과학적 근거
생활 문제 적용
```

---

### 3-2. 종단 연결

종단 연결은 현재 교과 개념이 다음 학년 과목이나 더 심화된 과목으로 이어지는 흐름이다.

예시:

```text
통합과학1: 과학의 측정과 우리 사회
→ 수학: 대수, 확률과 통계
→ 정보: 데이터 처리, 알고리즘
→ 지구과학: 기후·대기 자료 해석
→ 물리학: 측정 장치, 시스템 해석
```

4번 후속 연계축은 기본적으로 이 종단 연결을 보여주는 장치다.

즉, 4번은 단순히 학과를 맞추는 영역이 아니라:

```text
현재 교과 개념이 다음 학년 과목과 어떤 탐구축으로 이어지는지 보여주는 영역
```

이다.

다만 학과 입력은 그 축의 우선순위를 조정한다.

---

### 3-3. 현재 4번의 역할

4번의 정확한 역할은 아래와 같다.

```text
교과 개념 기반 종단 연결
+
학과 기반 우선순위 보정
+
보고서 방향의 사고축 결정
```

따라서 4번은 다음 세 가지를 동시에 만족해야 한다.

```text
1. 교과 개념에서 출발해야 한다.
2. 다음 과목으로 이어지는 종단성을 보여야 한다.
3. 학과에 따라 1순위 축과 설명 문장이 달라져야 한다.
```

## 4. 현재 화면 흐름의 절대 유지 기준

다음 화면 흐름은 유지해야 한다.

```text
상단 입력 영역
3. 교과 개념·키워드 선택
4. 교과 개념 기반 후속 연계축 선택

8. 보고서 라인 선택
학생용 결과 생성
MINI/Worker 생성 결과 출력
```

절대 하면 안 되는 것:

```text

- 3번 교과 개념과 4번 후속 연계축을 건너뛰기
- 도서를 마지막 카드에만 반영하기
- 결과 생성부만 반복 패치하기
```

## 5. 기존 v34 문서와 현재 v58 문서의 관계

기존 문서에서 “3번·4번은 잠금 상태”라고 되어 있다.

이 문장의 의미는 다음과 같이 해석해야 한다.

```text
3번·4번의 화면 구조와 개념은 새로 만들지 않는다.
```

하지만 이번 v58 검수 결과, 4번의 **학과별 축 판정과 우선순위 로직**에는 수정이 필요하다.

즉:

```text
4번 구조는 유지한다.
4번 로직은 정밀 보정한다.
```

잘못된 해석:

```text
4번은 어떤 문제가 있어도 절대 수정하지 않는다.
```

올바른 해석:

```text
4번 화면 구조는 유지하되,
학과 bucket, axisDomain, 직접/브리지/보조 판정은 보정한다.
```

## 6. v53~v58 진행 요약

### v53

보고서 기본 구조가 잡힌 상태였다.
결과 화면에서 질문을 내 사례로 바꾸고, 자료를 문단에 배치하고, 표와 결론으로 완성하는 흐름이 보였다.

하지만 결과값 다양성은 아직 부족했다.

---

### v54

목적:

```text
학과별 결과 렌즈 강화
```

대표 기준:

```text
컴퓨터공학과 → 입력값·조건문·오류 검증
환경공학과 → 환경 변수·노출·취약성·저감 기준
도시공학과 → 공간 구조·도시 열섬·녹지·인프라
간호학과 → 생체 반응·위험 요인·관리 기준
경영학과 → 비용·편익·의사결정·위험 관리
```

검수 결과:

```text
학과 렌즈가 일부 들어갔지만, 도서나 학과 변화가 전체 보고서 구조를 충분히 바꾸지는 못했다.
```

---

### v55

목적:

```text
도서 선택값을 마지막 카드뿐 아니라 제목·질문·자료 배치·결론에 반영
```

예시:

```text
부분과 전체
→ 부분 자료 하나만으로 전체를 판단해도 되는가?

침묵의 봄
→ 환경 변화가 생활 피해로 어떻게 확산되는가?
```

검수 결과:

```text
도서 카드 쪽은 개선되었지만, 도서가 보고서 전체 흐름을 바꾸는 정도는 아직 약했다.
```

---

### v56

목적:

```text
6번 전개 방식, 7번 관점, 8번 라인을 MINI payload와 결과 구조에 직접 반영
```

추가된 개념:

```text
reportMode
reportView
reportLine
reportChoiceBlueprint
choiceSummary
modeLabel
viewLabel
lineLabel
tableRows
paragraphRule
```

검수 결과:

```text
6·7·8 선택값이 결과에 반영되기 시작했으나,
학생 화면에 내부 선택명이 그대로 노출되는 문제가 생겼다.
```

문제 예시:

```text
6번 전개 방식
7번 관점
8번 라인
데이터 확장형
자료 해석 관점
```

---

### v57

목적:

```text
내부 선택명을 학생용 문장으로 번역
```

예상 변화:

```text
데이터 확장형
→ 수치·그래프·자료 출처를 추가해 판단 기준을 넓힌다.

자료 해석 관점
→ 기준 자료 / 실제 자료 / 내 해석 중심으로 정리한다.

확장형
→ 공식 기준 + 실제 자료 + 비교·검증 자료 구조로 확장한다.
```

검수 결과:

```text
내부 선택값 노출은 줄었지만,
학과값에 따른 결과 차이는 여전히 약했다.
```

---

### v58

목적:

```text
학생용 표현 보정 + 학과별 결과 차이 강화
```

기대 방향:

```text
컴퓨터공학과
→ 입력값, 조건문, 판단 결과, 오류 가능성

환경공학과
→ 환경 변수, 노출 시간, 취약성, 위험도, 저감 기준

도시공학과
→ 공간 구조, 도시 열섬, 녹지, 포장 면적, 생활권 차이
```

검수 결과:

```text
학과를 바꿨는데도 결과가 이전 학과의 후속 연계축에 끌리는 문제가 남았다.
```

결론:

```text
문제는 결과 생성부가 아니라 4번 후속 연계축 산출 단계에 있다.
```

## 7. 현재 핵심 문제

현재 발견된 핵심 문제는 아래다.

```text
학과를 바꿨는데 4번 후속 연계축의 직접/브리지/보조 판정이 정확히 바뀌지 않는다.
```

특히 다음 문제가 확인되었다.

```text
1. 컴퓨터공학과에서 여러 축이 모두 “바로 이어지는 축”으로 뜬다.
2. 환경공학과는 비교적 정상이나, 일부 보조축 설명이 과하게 직접 연결로 잡힌다.
3. 도시공학과는 폭염주의보 문맥에서 물리·시스템 해석 축이 1순위로 잡히는 문제가 있다.
4. “데이터”라는 단어 때문에 컴퓨터공학과 직접 연결로 과도하게 판정되는 경향이 있다.
5. 도시공학과가 urban/spatial bucket으로 충분히 분리되지 않는다.
```

## 8. 4번 후속 연계축 검수 결과

검수 고정 조건:

```text
과목: 통합과학1
교과 개념: 과학의 측정과 우리 사회
핵심 키워드: 폭염주의보
```

학과만 바꿔서 확인했다.

---

### 8-1. 컴퓨터공학과

현재 상태:

```text
1순위: 수리·데이터 모델링 축
물리·시스템 해석 축: 직접 연계 강함
지구·환경 데이터 해석 축: 직접 연계 강함
```

판단:

```text
수리·데이터 모델링 축이 1순위인 것은 맞다.
하지만 다른 축까지 모두 컴퓨터공학과와 바로 이어지는 축으로 뜨는 것은 과하다.
```

정상 기대값:

```text
수리·데이터 모델링 축
→ 직접 연계 강함

물리·시스템 해석 축
→ 역량 브리지

지구·환경 데이터 해석 축
→ 보조 확장 또는 역량 브리지
```

---

### 8-2. 환경공학과

현재 상태:

```text
1순위: 지구·환경 데이터 해석 축
2순위: 물리·시스템 해석 축
3순위: 수리·데이터 모델링 축
```

판단:

```text
지구·환경 데이터 해석 축이 1순위인 것은 정상 방향이다.
```

보완 기대값:


### 8-3. 도시공학과

판단:

```text
1순위: 지구·환경 데이터 해석 축
```

## 12. 학과 bucket 감지 규칙

도시공학과는 환경이나 공학 일반보다 먼저 검사해야 한다.

```javascript
function detectMajorBucket(text) {
  const t = String(text || "");

  if (/(컴퓨터|소프트웨어|인공지능|AI|데이터|정보|보안|프로그래밍)/.test(t)) {

    return "urban";

  if (/(환경|기후|대기|지구|해양|생태|자원|에너지환경|환경공학)/.test(t)) {
    return "env";

    return "business";
  }

}
```

## 17. 학과 검색 버퍼링 문제

사용자가 학과 검색 시 버퍼링이 있다고 보고했다.

의심 원인:

```text
학과 검색 입력 중 매 keyup/change마다 3~8번 영역 전체가 재렌더링됨
학과 후보 검색, 학과 선택, 4번 축 재계산이 분리되어 있지 않음
검색 결과 캐시 또는 debounce/throttle이 부족함
```

수정 방향:

```text
1. 학과 검색 입력 중에는 후보 리스트만 갱신
2. 실제 학과를 선택했을 때만 4번 이후 영역 재계산
3. 검색어별 결과 캐시 적용
4. input 이벤트에는 debounce 120~200ms 적용
5. renderTrackArea, renderBookArea, renderReportChoiceArea는 학과 확정 이후에만 실행
```

예시:

```javascript
let majorSearchTimer = null;
const majorSearchCache = new Map();

function onMajorSearchInput(q) {
  clearTimeout(majorSearchTimer);
  majorSearchTimer = setTimeout(() => {
    renderMajorSearchCandidates(q);
  }, 150);
}

function renderMajorSearchCandidates(q) {
  const key = String(q || "").trim();
  if (majorSearchCache.has(key)) {
    renderMajorCandidates(majorSearchCache.get(key));
    return;
  }

  const results = searchMajors(key);
  majorSearchCache.set(key, results);
  renderMajorCandidates(results);
}

function onMajorConfirmed(majorName) {
  const prev = state.majorSelectedName;

  state.majorSelectedName = majorName;
  state.career = majorName;
  state.majorProfile = buildMajorRoutingProfile(majorName);

  if (prev !== majorName) {
    clearAfterMajorChange();
  }

  renderTrackArea();
}
```
