## 2. 절대 유지해야 할 작업 원칙

### 2-1. 실제 데이터에 있는 학과명만 사용

학과명은 반드시 ZIP 내부 또는 실제 화면에 존재하는 학과명만 사용한다.

```text
금지:
임의 학과명
실제 데이터에 없는 예시 학과명
화면에 없는 학과명
```

이번 채팅에서 반복된 문제는 **학과명은 맞았지만 3번·4번을 실제 화면 기준이 아니라 계열 추론으로 잡은 것**이었다.

앞으로는 반드시 아래 순서를 지킨다.

```text
1. 실제 학과명 확인
2. 실제 과목 확인
3. 실제 3번 추천 개념 확인
4. 실제 4번 후속 연계축 확인
5. 그다음 도서 기대값 제시
```

---

### 2-2. 3번 추천 개념은 실제 화면 기준만 사용

절대 계열 추론으로 잡지 않는다.

예를 들어 어문계라고 해서 무조건 아래처럼 잡으면 안 된다.

```text
서정 갈래와 시적 표현
서사·극 갈래와 이야기 구성
교술 갈래와 성찰적 표현
```

실제로 아랍어과, 한국어학과는 화면에서 아래처럼 나왔다.

```text
비판적 읽기와 토론
사회적 쟁점 글쓰기와 문장 구성
공동체 의사소통과 공감
```

따라서 새 채팅에서는 반드시 사용자가 올린 화면 또는 실제 엔진 데이터를 기준으로 잡는다.

---

### 2-3. 4번 후속 연계축은 카드 제목 우선

4번 카드의 설명문 안에 들어 있는 단어를 읽고 오작동하면 안 된다.

예시 문제:

```text
표현·창작 확장 축 설명문 안에 “매체 표현”이 들어 있음
→ 기존 로직이 정서·매체 변환 축으로 오인
```

해결 기준:

```text
4번 판별은 카드 설명문이 아니라 카드 제목/선택값 우선
```

---

### 2-4. 패치는 항상 최소 파일만

사용자 선호:

```text
전체 파일보다 적용 파일만 담은 ZIP 선호
파일명은 짧고 영문 버전명 사용
```

기본 제공 방식:

```text
book_210_ui_bridge_v206_patch.zip
└─ assets/js/book_210_ui_bridge.js
```

건드리면 안 되는 것:

```text
ACCESS_GATEWAY
mini_worker_generate_bridge_v32.js
keyword_engine.js
보고서 생성 구조
UI 구조
학과 라우팅 구조
major_search_fast_guard.js 재추가 금지
```

### v205: 한문학과

현재 마지막 패치 제공 상태.

중요:

```text
한문학과는 아직 사용자가 “완성이야”라고 검수 완료를 말하지 않았다.
```

따라서 새 채팅 첫 작업은 한문학과 검수부터 해야 한다.

제공된 기준:

```text
학과: 한문학과
과목: 공통국어1
대표 확인: 비판적 읽기와 토론
```

도서 기대값:

```text
자료 검증·쟁점 분석 축
→ 논어
→ 맹자
→ 성호사설

논증·토론 축
→ 맹자
→ 논어
→ 국가

비판 해석 확장 축
→ 성호사설
→ 사기열전
→ 논어
```

보조 확인:

```text
한문학과 / 공통국어1 / 서사·극 갈래와 이야기 구성

정상 기대 도서:
사기열전
삼국유사
논어
```

## 6. 새 채팅에서 바로 시작할 문장

아래를 새 채팅 첫 메시지로 넣는다.

```text
수행평가 탐구엔진 keyword-engine 5번 도서 추천 로직 검수 작업을 이어서 진행한다.

현재 최신 패치 버전은 v205이고, 수정 파일은 assets/js/book_210_ui_bridge.js이다.

현재까지 완료된 학과:
경영학과, 경제학과, 사회학과, 행정학과, 정치외교학과, 법학과,
미디어커뮤니케이션학과, 국어국문학과, 심리학과, 교육학과,
사회복지학과, 문화콘텐츠학과, 광고홍보학과, 사학과,
고고학과, 문화인류학과, 문화유산학과, 문예창작학과,
철학과, 미학과, 문헌정보학과, 영어영문학과, 일어일문학과,
중어중문학과, 독어독문학과, 불어불문학과, 노어노문학과,
아랍어과, 언어학과, 한국어학과.

현재 검수 대기:
한문학과 v205.

절대 규칙:
1. 학과명은 반드시 업로드한 keyword-engine.zip 내부 실제 데이터에 존재하는 학과명만 사용한다.
2. 3번 교과 개념은 실제 화면에 뜨는 값만 사용한다.
3. 4번 후속 연계축은 카드 설명문이 아니라 카드 제목/선택값을 우선 판별한다.
4. 임의 학과명, 임의 교과 개념명, 임의 축명을 예시로도 사용하지 않는다.
5. 계열 묶음 패치 시에는 대표 학과 1개만 전체 확인하고, 나머지는 실제 존재 학과명 기준으로 핵심 축 2~3개만 표본 확인한다.
6. 전체 파일이 아니라 적용 파일만 담은 패치 ZIP으로 제공한다.
7. ZIP 내부 경로는 assets/js/book_210_ui_bridge.js로 유지한다.
8. ACCESS_GATEWAY, mini worker, 보고서 생성 구조, UI 구조는 건드리지 않는다.
9. major_search_fast_guard.js는 재추가하지 않는다.
10. 학과 후보 카드 UI로 변경하지 않는다.

먼저 한문학과 v205 검수부터 이어가고, 완료되면 실제 데이터에 남은 학과 목록을 확인한 뒤 다음 대표 학과 1개를 제시해줘.
```
