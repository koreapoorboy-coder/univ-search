# DATABASE_1_TO_5_LOCK_2026-05-12

## keyword-engine 1~5번 1차 잠금 통합 DB

작성일: 2026-05-12  
목적: `keyword-engine` 수행평가 탐구엔진의 **1~5번 흐름 1차 완료 상태**를 하나의 DB/인계 파일로 고정한다.  
적용 기준: 사용자가 검수 완료라고 확인한 현재 상태 기준.  
이 파일은 기존 `DATABASE.MD`와 `0512.md`의 핵심 기준을 포함하고, 이번 채팅에서 완료된 v210~v212 도서 매핑 잠금 및 v213~v214 진행 상황까지 반영한 통합본이다.

# 0. 새 채팅 즉시 시작용 문장

새 채팅에서는 아래 문장을 그대로 먼저 넣고, 최신 `keyword-engine.zip` 또는 현재 적용본을 함께 올리면 된다.

```text
수행평가 탐구엔진 keyword-engine 작업을 이어서 진행한다.
이 채팅의 기준 DB는 DATABASE_1_TO_5_LOCK_2026-05-12.md이다.

현재 1~5번은 1차 완료 상태로 잠금한다.
1번 희망 진로/학과 검색, 2번 과목 선택, 3번 교과 개념/추천 키워드, 4번 후속 연계축, 5번 도서 매핑은 큰 오류 제거와 대표/표본 검수를 마친 상태다.

앞으로는 1~5번 구조를 새로 갈아엎지 말고, 2차 업그레이드 기준으로만 작업한다.
2차 업그레이드의 우선순위는 다음이다.
1. 5번 선택 도서가 mini payload에 정확히 들어가는지 확인
2. 선택 도서의 추천 이유/활용 방식이 보고서에 실제 반영되는지 확인
3. 6~8번 보고서 전개 방식/관점/라인이 단순 word가 아니라 MINI 작성 지시문으로 들어가는지 검수
4. 탐구 실행 지도 생성 오류가 나면 v214의 /__mini/generate fallback 구조를 기준으로 점검

절대 규칙:
- 실제 ZIP 내부 또는 실제 화면에 존재하는 학과명만 사용한다.
- 실제 화면에 뜨는 과목명, 3번 교과 개념명, 4번 후속 연계축명만 사용한다.
- 계열 추론으로 3번·4번·도서 기대값을 만들지 않는다.
- 4번 판별은 카드 설명문이 아니라 카드 제목/선택값을 우선한다.
- 패치는 최소 파일만 제공한다.
- major_search_fast_guard.js는 재추가하지 않는다.
- 학과 후보 카드 UI로 변경하지 않는다.
- ACCESS_GATEWAY와 mini worker는 필요한 경우 외에는 건드리지 않는다.
- 기존 1~5번 잠금 구조를 깨지 않는다.
```

# 2. 프로젝트 마스터 원칙 요약

이 엔진은 단순 키워드 추천기나 주제 생성기가 아니다. 핵심은 다음 흐름이다.

```text
희망 진로/학과
→ 과목 선택
→ 교과 개념 선택
→ 추천 키워드 선택
→ 교과 개념 기반 후속 연계축 선택
→ 도서 추천
→ 보고서 방향 설계
→ mini payload 전달
→ 최종 보고서 생성
```

엔진의 정체성은 다음 문장으로 고정한다.

```text
이 엔진은 전공 키워드 추천기가 아니라,
교과 개념에서 출발해 횡단·종단·교과 간 연결을 만들어주는 수행평가 탐구 네비게이션이다.
```

따라서 모든 패치와 DB 보정은 다음 기준을 지켜야 한다.

```text
전공 관심사 → 교과 개념 → 추천 키워드 → 후속 연계축 → 도서 → 보고서
```

전공 키워드가 먼저 오면 안 된다. 예를 들어 이차전지, AI, 반도체, 의학 같은 키워드는 최종 목적지 또는 확장 방향일 뿐이며, 학생 화면과 보고서 설계는 반드시 고등학교 교과 개념에서 출발해야 한다.

# 3. 1~5번 단계별 DB 정의

## 3-1. 1번: 희망 진로/학과 검색

### 역할

학생이 희망 진로 또는 학과를 입력하면, 실제 데이터에 존재하는 학과만 선택 후보로 제공한다.

### 잠금 원칙

```text
- 실제 ZIP 내부에 존재하는 학과명만 사용한다.
- 실제 화면에 없는 학과명을 예시로 들지 않는다.
- 임의 학과명, 임의 계열명, 임의 전공명 생성 금지.
- 학과 후보 카드 UI로 변경하지 않는다.
- major_search_fast_guard.js 재추가 금지.
```

### 2차 검수 포인트

```text
- 사용자가 검색한 학과명이 실제 데이터 학과로 매칭되는가?
- 비슷한 이름 학과가 있을 때 잘못된 학과로 자동 이동하지 않는가?
- 선택한 학과명이 mini payload의 department 또는 major 계열 필드에 유지되는가?
```

---

## 3-2. 2번: 과목 선택

### 역할

선택 학과에 맞는 고교 과목을 추천하고, 학생이 실제 수업/탐구에 사용할 과목을 선택하게 한다.

### 잠금 원칙

```text
- 실제 엔진 데이터의 과목명만 사용한다.
- 과목명은 교육과정 기준에 맞는 명칭을 유지한다.
- 학과별 추천 과목은 계열 추론이 아니라 현재 데이터와 화면 기준으로 판단한다.
```

### 2차 검수 포인트

```text
- 학과 선택 후 과목 카드가 과도하게 버퍼링되지 않는가?
- 과목 선택 후 3번 교과 개념이 즉시 갱신되는가?
- 과목 선택값이 mini payload에 들어가는가?
```

---

## 3-3. 3번: 교과 개념/추천 키워드 선택

### 역할

3번은 보고서의 출발점이다. 학생은 이 단계에서 교과 개념을 먼저 고르고, 그다음 추천 키워드를 선택한다.

### 잠금 원칙

```text
- 3번 교과 개념은 실제 화면 기준만 사용한다.
- 계열 추론으로 3번을 만들지 않는다.
- 학과명만 맞고 3번이 틀리는 오류를 최우선으로 경계한다.
- 사용자가 올린 캡처 또는 실제 엔진 화면을 기준으로 잠금한다.
```

### 반복 오류 사례

```text
어문계라고 해서 모두 아래 문학 갈래형으로 잡으면 안 된다.
- 서정 갈래와 시적 표현
- 서사·극 갈래와 이야기 구성
- 교술 갈래와 성찰적 표현

아랍어과, 한국어학과는 실제 화면에서 아래처럼 나왔었다.
- 비판적 읽기와 토론
- 사회적 쟁점 글쓰기와 문장 구성
- 공동체 의사소통과 공감
```

### 2차 검수 포인트

```text
- 실제 화면에 뜨는 3개 기본 개념과 패치 기대값이 일치하는가?
- 추천 키워드가 왼쪽 교과 개념 클릭 전에는 대기 상태로 남아 있는가?
- 교과 개념을 클릭한 뒤 추천 키워드가 정상 갱신되는가?
- 추천 키워드 선택값이 4번과 5번으로 전달되는가?
```

---

## 3-4. 4번: 교과 개념 기반 후속 연계축

### 역할

4번은 보고서의 탐구 방향을 결정한다. 단순 태그가 아니라 보고서의 질문과 분석 방식, 도서 추천 방향을 바꾸는 핵심 축이다.

### 잠금 원칙

```text
- 4번 후속 연계축 판별은 카드 설명문이 아니라 카드 제목/선택값 우선.
- 설명문 안의 단어 때문에 다른 축으로 오인하지 않는다.
- 4번은 정답 1개가 아니라 학생 선택 후보군이다.
- 1순위처럼 보이는 표현은 최소화한다.
```

### 반복 오류 사례

```text
표현·창작 확장 축 설명문 안에 “매체 표현”이 들어 있음
→ 기존 로직이 정서·매체 변환 축으로 오인
→ 해결: 카드 제목/선택값 우선 판별
```

### 2차 검수 포인트

```text
- 3번 교과 개념을 바꾸면 4번 축도 정상 갱신되는가?
- 빈 공간 클릭 시 4번 축이 임의로 바뀌지 않는가?
- 4번 축 선택값이 도서 추천과 mini payload에 전달되는가?
- 카드 제목과 payload 값이 서로 어긋나지 않는가?
```

---

## 3-5. 5번: 직접 일치/확장 참고 도서 추천

### 역할

5번은 독후감용 도서 추천이 아니다. 도서는 탐구 보고서 안에서 근거, 관점, 확장성을 제공해야 한다.

도서 추천은 다음 구조로 작동해야 한다.

```text
교과 개념
→ 추천 키워드
→ 후속 연계축
→ 직접 도서
→ 확장 도서
→ 보고서 활용 방식
```

### 직접 도서와 확장 도서

| 구분 | 의미 |
|---|---|
| 직접 도서 | 교과 개념과 탐구 주제를 직접 설명하거나 분석하는 데 쓰는 책 |
| 확장 도서 | 사회적 의미, 윤리, 진로, 한계, 쟁점으로 확장하는 책 |

### 5번 잠금 원칙

```text
- 같은 계열에 같은 책만 반복 추천되는 문제를 제거한다.
- 전공/교과/후속축과 무관한 책이 메인으로 뜨지 않게 한다.
- 확장 도서를 직접 도서처럼 고정하지 않는다.
- 선택한 후속축과 도서 추천이 연결되어야 한다.
- 추천 이유와 활용 방식은 2차 업그레이드에서 정밀화한다.
```

### 2차 검수 포인트

```text
- 책이 화면에 뜨는가?
- 학생이 책을 선택할 수 있는가?
- 선택한 책 제목이 payload에 들어가는가?
- 추천 이유가 payload에 들어가는가?
- 보고서 활용 방식이 payload에 들어가는가?
- 최종 보고서에서 선택한 책이 실제로 언급되는가?
```

# 5. 이번 채팅 핵심 수정 DB

## 5-1. 한문학과 v210 잠금

### 문제

인계문 기준에서는 한문학과가 `비판적 읽기와 토론` 계열로 제시되어 있었으나, 사용자가 올린 실제 화면에서는 아래 3개 교과 개념이 나왔다.

```text
학과: 한문학과
과목: 공통국어1

실제 3번 교과 개념:
- 서정 갈래와 시적 표현
- 서사·극 갈래와 이야기 구성
- 교술 갈래와 성찰적 표현
```

### 처리

```text
v209의 비판적 읽기·토론형 기준 제거.
현재 화면에 실제 존재하는 문학 갈래형 3개 개념만 반영.
한문학과 A-48 하드락을 v210 기준으로 수정.
```

### 대표 확인 1

```text
한문학과 / 공통국어1 / 서정 갈래와 시적 표현

기대 도서:
문학 감상·해석 축
→ 논어
→ 맹자
→ 성호사설
```

## 7-4. v214 생성 오류 보정

탐구 실행 지도 생성 시 아래 에러가 발생했다.

```text
405 Not Allowed
HTML 응답 노출
```

해석:

```text
/__mini/generate 주소로 POST 요청이 갔지만,
현재 접속 주소가 해당 요청을 받지 못해 HTML 405 페이지가 반환된 상태.
```

v214 기준 보정:

```text
1순위 생성 주소는 /__mini/generate 유지.
405 / 404 / HTML 오류가 나오면 기존 Worker generate 주소로 fallback.
405 HTML이 화면에 그대로 뜨지 않도록 오류 메시지 정리.
```

주의:

```text
운영용 access-gateway 주소에서는 /__mini/generate가 먼저 작동해야 한다.
GitHub Pages나 정적 주소 테스트 중이면 fallback이 필요하다.
```

# PROJECT_MASTER_CONTEXT

## keyword-engine 수행평가 탐구엔진 전체 구조·방향성·작업 원칙

## 0. 이 문서의 목적

이 문서는 `keyword-engine` 프로젝트의 전체 구조와 방향성을 고정하기 위한 기준 문서다.

이 문서는 단순 작업 로그가 아니다.
이 엔진이 어떤 목적을 가지고 만들어졌는지, 어떤 방식으로 학생의 탐구를 이끌어야 하는지, 그리고 앞으로 어떤 파일을 수정하더라도 절대 잃어버리면 안 되는 핵심 원칙을 정리한 문서다.

이 문서는 새 채팅에서 작업을 이어갈 때 반드시 먼저 읽고 적용해야 한다.

---

# 1. 엔진의 정체성

이 엔진은 단순한 키워드 추천기나 주제 생성기가 아니다.

이 엔진의 목적은 학생이 다음 흐름으로 **수행평가 탐구 보고서의 방향을 스스로 설계하도록 돕는 탐구 네비게이션**이다.

핵심 구조는 다음과 같다.

```text
희망 진로/학과
→ 과목 선택
→ 교과 개념 선택
→ 추천 키워드 선택
→ 교과 개념 기반 후속 연계축 선택
→ 도서 추천
→ 보고서 방향 설계
→ mini payload 전달
→ 최종 보고서 생성
```

이 엔진은 “정답 주제”를 하나 던지는 도구가 아니다.

학생이 선택한 교과 개념과 진로 방향을 바탕으로,
**어떤 교과 개념에서 출발해 어떤 탐구 흐름으로 확장할 수 있는지**를 안내하는 도구다.

따라서 이 엔진의 핵심은 다음 문장으로 요약된다.

```text
이 엔진은 목적지를 대신 정해주는 도구가 아니라,
학생이 교과 개념에서 출발해 탐구 방향을 선택하도록 돕는 네비게이션이다.
```

---

# 2. 가장 중요한 설계 철학

## 2-1. 전공 키워드보다 교과 개념이 먼저다

이 엔진은 “이차전지”, “AI”, “반도체”, “의학” 같은 전공 키워드에서 바로 출발하면 안 된다.

반드시 고등학교 교과 개념에서 출발해야 한다.

예를 들어 학생이 이차전지에 관심이 있어도 바로 “이차전지 보고서”를 추천하는 것이 아니라, 아래처럼 교과 개념으로 내려와야 한다.

```text
이차전지
→ 산화 환원 반응
→ 전자 이동
→ 전위차
→ 에너지 변환
→ 배터리 효율 또는 소재 안정성 탐구
```

즉, 전공 키워드는 최종 목적지가 아니라 **교과 개념을 확장하는 방향성**이다.

엔진은 항상 다음 구조를 지켜야 한다.

```text
전공 관심사 → 교과 개념 → 추천 키워드 → 후속 연계축 → 탐구 보고서
```

---

## 2-2. 학생 선택형 구조여야 한다

이 엔진은 자동으로 하나의 주제를 정해주면 안 된다.

학생은 다음 단계마다 선택해야 한다.

```text
과목 선택
→ 교과 개념 선택
→ 추천 키워드 선택
→ 후속 연계축 선택
→ 도서 선택
```

따라서 화면에서 “1순위”처럼 정답처럼 보이는 표현은 최소화해야 한다.

이미 4번 후속 연계축의 `1순위` 라벨은 제거했다.
이 판단은 유지해야 한다.

이유는 다음과 같다.

```text
후속 연계축은 정답이 아니라 후보군이다.
학생이 여러 후보 중 자신의 탐구 방향에 맞는 축을 선택해야 한다.
```

---

## 2-3. 보고서 결과보다 탐구 경로가 중요하다

이 엔진은 완성된 보고서 문장을 바로 제공하는 것이 목적이 아니다.

더 중요한 것은 학생이 다음 질문에 답할 수 있도록 만드는 것이다.

```text
나는 어떤 교과 개념에서 출발했는가?
왜 이 키워드를 선택했는가?
이 키워드는 어떤 후속 탐구축으로 이어지는가?
이 도서는 내 탐구에 어떤 근거가 되는가?
내 보고서는 어떤 흐름으로 전개되는가?
```

따라서 보고서 생성은 단순 문장 생성이 아니라,
**학생의 선택 경로를 반영한 탐구 구조화**여야 한다.

# 3. 핵심 연결 구조

이 엔진에서 가장 중요한 연결 구조는 세 가지다.

```text
1. 횡단 연결
2. 종단 연결
3. 교과 간 연결
```

이 세 가지를 잃으면 이 엔진은 단순 주제 추천기가 된다.

---

# 4. 횡단 연결 구조

## 4-1. 횡단 연결의 의미

횡단 연결은 같은 시기 또는 같은 학년 수준에서 여러 과목의 개념이 서로 연결되는 구조다.

예를 들어 고1 학생이 `통합과학`에서 배운 내용을 `수학`, `정보`, `통합사회`와 연결할 수 있어야 한다.

예시:

```text
통합과학: 기후 변화
→ 수학: 자료 해석, 변화율, 그래프
→ 정보: 데이터 시각화
→ 통합사회: 지속가능한 삶, 정책 선택
→ 보고서: 기후 데이터 분석과 사회적 대응 방안
```

이런 구조가 횡단 연결이다.

---

## 4-2. 횡단 연결이 필요한 이유

학생부와 수행평가에서 중요한 것은 활동 개수가 아니다.
중요한 것은 학생의 사고가 과목 사이에서 어떻게 확장되는가이다.

따라서 엔진은 다음을 유도해야 한다.

```text
하나의 교과 개념이 다른 과목의 개념과 자연스럽게 연결되는가?
학생이 같은 문제를 여러 교과의 관점으로 바라볼 수 있는가?
보고서 안에서 수학적 자료 해석, 과학적 원리, 사회적 의미가 연결되는가?
```

---

## 4-3. 횡단 연결 예시

### 예시 1. 반도체공학과

```text
화학: 화학 결합
→ 물리: 물질의 전기적 특성
→ 정보: 신호 처리와 데이터 표현
→ 수학: 지수·로그 모델
→ 탐구 방향: 반도체 소재의 전기적 특성과 신호 해석
```

### 예시 2. 생명공학과

```text
생명과학: 유전자와 염색체
→ 화학: 탄소 화합물의 구조
→ 정보: 염기서열 데이터
→ 수학: 확률과 통계
→ 탐구 방향: 유전 정보와 생체 분자의 구조적 해석
```

### 예시 3. 경영학과

```text
통합사회: 시장 경제와 금융 생활
→ 확률과 통계: 통계적 추정
→ 정보: 자료와 정보의 분석
→ 수학: 함수와 변화율
→ 탐구 방향: 소비자 데이터 기반 시장 의사결정
```

---

# 5. 종단 연결 구조

## 5-1. 종단 연결의 의미

종단 연결은 학년 또는 과목 위계에 따라 개념이 깊어지는 구조다.

즉, 학생이 1학년에서 배운 개념이 2학년, 3학년 선택과목으로 어떻게 이어지는지를 보여줘야 한다.

예시:

```text
통합과학1: 물질 구성과 분류
→ 화학: 원소의 주기적 성질
→ 물질과 에너지: 분자 간 힘과 물성
→ 고급 탐구: 소재 물성 분석
```

이런 흐름이 종단 연결이다.

---

## 5-2. 종단 연결이 필요한 이유

종단 연결이 있어야 학생의 탐구가 단발성 활동이 아니라 성장 흐름으로 보인다.

학생부 평가에서 중요한 것은 다음이다.

```text
1학년 때 무엇을 궁금해했는가?
2학년 때 그 궁금증을 어떤 과목에서 심화했는가?
3학년 때 어떤 탐구로 완성했는가?
```

따라서 엔진은 한 번의 보고서 주제를 추천하는 데서 끝나면 안 된다.
해당 주제가 이후 어떤 과목과 활동으로 이어질 수 있는지를 보여줘야 한다.

---

## 5-3. 종단 연결 예시

### 예시 1. 신소재공학과

```text
통합과학1: 물질 구성과 분류
→ 화학: 화학 결합
→ 물질과 에너지: 액체의 물성과 분자 간 힘
→ 전자기와 양자: 양자와 물질의 상호작용
→ 탐구 완성: 소재의 구조와 물성 변화 분석
```

### 예시 2. 의예과

```text
생명과학: 면역과 백신
→ 세포와 물질대사: 세포의 구조와 물질 이동
→ 화학: 화학 반응에서의 동적 평형
→ 탐구 완성: 체액 pH와 면역 반응의 항상성 탐구
```

### 예시 3. 데이터사이언스학과

```text
대수: 지수함수와 로그함수
→ 확률과 통계: 확률변수와 확률분포
→ 정보: 자료와 정보의 분석
→ 미적분1: 도함수의 활용
→ 탐구 완성: 데이터 변화 모델링과 예측 분석
```

# 6. 교과 간 연결 구조

## 6-1. 교과 간 연결의 의미

교과 간 연결은 단순히 “여러 과목을 나열하는 것”이 아니다.

각 과목의 개념이 보고서 안에서 역할을 가져야 한다.

예를 들어 다음은 약한 연결이다.

```text
이 주제는 과학, 수학, 사회와 관련이 있다.
```

이건 의미가 없다.

강한 연결은 아래처럼 역할이 분명해야 한다.

```text
과학: 현상의 원리 설명
수학: 자료 해석과 모델링
정보: 데이터 처리와 시각화
사회: 문제의 영향과 해결 방향 논의
```

---

## 6-2. 엔진에서 교과 간 연결이 작동해야 하는 방식

엔진은 후속축과 도서 추천, 보고서 생성에서 다음 역할 분리를 반영해야 한다.

| 교과 | 보고서 안에서의 역할            |
| -- | ---------------------- |
| 과학 | 원리, 현상, 실험, 인과 구조      |
| 수학 | 모델링, 계산, 그래프, 변화율, 확률  |
| 정보 | 데이터 처리, 알고리즘, 시각화, 시스템 |
| 사회 | 정책, 윤리, 경제, 공동체 영향     |
| 국어 | 자료 해석, 논증 구조, 표현력      |
| 영어 | 자료 탐색, 해외 사례, 원문 이해    |

---

# 7. 추천 키워드의 역할

추천 키워드는 단순 검색어가 아니다.

추천 키워드는 학생이 같은 교과 개념 안에서 어떤 방향으로 탐구할지 고르는 장치다.

예를 들어 `화학 결합` 안에서도 추천 키워드 성격은 나뉜다.

| 키워드 유형 | 예시                  | 후속축 방향   |
| ------ | ------------------- | -------- |
| 구조형    | 공유 결합, 이온 결합, 전기음성도 | 결합 구조 해석 |
| 물성형    | 끓는점, 용해도, 전도성       | 물성 예측    |
| 응용형    | 반도체, 고분자, 의약품       | 전공 응용    |
| 데이터형   | 측정값, 그래프, 비교        | 자료 분석    |

1차 잠금에서는 키워드별 세부 차이를 완벽하게 나누지 않았다.
현재는 큰 방향 오류를 막는 데 집중했다.

2차 작업에서는 이 추천 키워드별 변별도를 봐야 한다.

---

# 8. 후속 연계축의 역할

4번 후속 연계축은 보고서의 방향을 결정하는 핵심이다.

후속축은 다음 질문에 답해야 한다.

```text
이 교과 개념을 어떤 방식으로 더 탐구할 것인가?
구조를 분석할 것인가?
데이터를 해석할 것인가?
실험을 설계할 것인가?
사회적 의미로 확장할 것인가?
전공 기술로 응용할 것인가?
```

따라서 후속축은 다음과 같은 기능을 한다.

| 기능              | 설명                  |
| --------------- | ------------------- |
| 탐구 방향 결정        | 보고서의 중심 질문을 결정      |
| 도서 추천 연결        | 어떤 도서가 필요한지 결정      |
| mini payload 전달 | 보고서 생성에 선택값 전달      |
| 보고서 문단 차별화      | 선택 축에 따라 분석 문단이 달라짐 |

---

# 9. 5번 도서 추천의 역할

5번 도서는 단순 독후활동이 아니다.

도서는 보고서 안에서 **근거, 관점, 확장성**을 제공해야 한다.

도서 추천은 다음 구조로 작동해야 한다.

```text
교과 개념
→ 추천 키워드
→ 후속 연계축
→ 직접 도서
→ 확장 도서
→ 보고서 활용 방식
```

---

## 9-1. 직접 도서와 확장 도서

도서는 반드시 두 종류로 구분해야 한다.

| 구분    | 의미                              |
| ----- | ------------------------------- |
| 직접 도서 | 교과 개념과 탐구 주제를 직접 설명하는 데 쓰는 책    |
| 확장 도서 | 사회적 의미, 윤리, 진로, 한계, 쟁점으로 확장하는 책 |

예시:

```text
컴퓨터공학과 + 데이터 예측 축
직접 도서: 데이터, 알고리즘, 모델링 관련 도서
확장 도서: 기술 윤리, 감시사회, AI와 인간 관련 도서
```

```text
의예과 + 면역과 백신 축
직접 도서: 면역, 질병, 생체 시스템 관련 도서
확장 도서: 의료 윤리, 공중보건, 감염병 정책 관련 도서
```

```text
환경공학과 + 기후·환경 영향 분석 축
직접 도서: 기후, 생태, 에너지, 탄소중립 관련 도서
확장 도서: 문명, 정책, 지속가능성, 사회적 책임 관련 도서
```

---

## 9-2. 도서 추천에서 피해야 할 것

5번 도서 추천에서 가장 조심해야 할 것은 다음이다.

```text
1. 모든 계열에 같은 책이 반복 추천되는 것
2. 확장 도서가 직접 도서처럼 고정되는 것
3. 전공과 무관한 책이 메인으로 뜨는 것
4. 추천 이유가 너무 추상적인 것
5. 선택한 후속축과 도서 추천이 연결되지 않는 것
```

예를 들어 `1984` 같은 책은 정보사회나 감시사회 맥락에서는 확장 도서로 가능하다.
하지만 컴퓨터공학과의 데이터 예측 탐구에서 직접 핵심 도서로만 고정되면 부적절하다.

# 10. mini payload의 역할

mini payload는 학생이 선택한 값을 최종 보고서로 넘기는 핵심 연결 장치다.

도서 추천까지 들어가면 반드시 아래 값들이 넘어가야 한다.

```text
department
subject
selectedConcept
selectedRecommendedKeyword
followupAxis
selectedBookTitle
selectedBookReason
selectedBookUsage
```

이 값이 빠지면 최종 보고서가 학생 선택을 반영하지 못하고 일반적인 문장으로 나온다.

따라서 5번 이후에는 반드시 다음을 확인해야 한다.

```text
책이 화면에 뜨는가?
학생이 책을 선택할 수 있는가?
선택한 책 제목이 payload에 들어가는가?
추천 이유가 payload에 들어가는가?
보고서 활용 방식이 payload에 들어가는가?
최종 보고서에서 그 책이 실제로 언급되는가?
```

# 11. 보고서 생성의 방향성

보고서는 단순히 그럴듯한 글이 나오면 안 된다.

보고서는 학생의 선택 경로를 반영해야 한다.

좋은 보고서 구조는 다음과 같다.

```text
1. 선택한 교과 개념 설명
2. 추천 키워드를 선택한 이유
3. 후속 연계축에 따른 탐구 질문
4. 도서를 통한 근거 또는 관점 확보
5. 자료 분석 또는 사례 적용
6. 한계와 추가 탐구 방향
```

보고서가 아래처럼 나오면 문제다.

```text
- 선택한 키워드가 반영되지 않음
- 후속축이 문단에 드러나지 않음
- 선택 도서가 보고서에 반영되지 않음
- 모든 학생에게 비슷한 문단이 나옴
- 전공 키워드만 있고 교과 개념이 없음
```

# 12. 작업 방식 원칙

## 12-1. 전체 교체보다 최소 패치

이 프로젝트는 이미 구조가 많이 쌓여 있다.

따라서 수정할 때는 전체 파일을 갈아엎지 말고, 문제 지점만 최소 패치해야 한다.

원칙:

```text
1. 문제를 먼저 화면 기준으로 확인
2. 관련 파일만 확인
3. 실제 존재하는 데이터명 기준으로 수정
4. 전체 구조 변경 금지
5. 패치 파일만 제공
6. 적용 후 검수
7. 정상 확인되면 잠금
```

---

## 12-2. 실제 화면 데이터명을 기준으로 작업

작업할 때 설명용 표현을 실제 데이터명처럼 쓰면 안 된다.

예를 들어 아래처럼 구분해야 한다.

| 설명용 표현       | 실제 화면 개념명      |
| ------------ | -------------- |
| 산염기 반응       | 화학 반응에서의 동적 평형 |
| 화학 반응의 양적 관계 | 물질의 양과 화학 반응식  |
| 산화 환원 반응     | 화학 반응과 열의 출입   |

항상 실제 엔진 화면에 존재하는 과목명, 개념명, 키워드명을 기준으로 작업해야 한다.

## 12-3. 1차와 2차를 구분

작업은 반드시 단계별로 나눠야 한다.

| 단계 | 목적                 |
| -- | ------------------ |
| 1차 | 계열별 큰 오류 제거        |
| 2차 | 추천 키워드별 변별도 조정     |
| 3차 | mini payload 반영 확인 |
| 4차 | 최종 보고서 문단 반영 확인    |

3번·4번은 현재 1차 완료 상태다.

다음 5번 도서도 동일하게 진행해야 한다.

```text
5번 1차: 도서 추천 큰 방향 잠금
5번 2차: 도서별 추천 이유와 변별도 보정

5번 4차: 최종 보고서 반영 확인
```

```text
4. 4번 설명문 안의 단어 때문에 다른 축으로 오인함
5. 기존 일반 도서 로직이 하드락보다 먼저 적용되어 과학 도서가 뜸
```
